# Lec10：语义分析进阶

## 1. 语义分析为何需要进阶

语义分析的任务，是在语法结构之上提取程序真正想表达的含义。在编译器流水线里，它位于词法分析和语法分析之后、主要优化与代码生成之前。实际工作通常包括把语法翻译成中间表示、进行类型检查，以及在某些场景下解释或推敲语言构造。

本讲一开始强调了一个很重要的区分：这里的“分析”不是优化器里的数据流分析。这里的核心任务是 **semantic elaboration**，也就是把语法层面的结构，翻译成更显式、更类型化、更适合后续编译阶段处理的表示。

![语义分析既可以在语法分析之后单独进行，也可以与语法分析交错进行](lec10_materials/semantic_elaboration_implementations.png)

可以把实现方式分成两类：

- 先完成语法分析，构造语法分析树或 AST，再独立遍历做语义分析；
- 在语法分析过程中同步执行语义分析，让语法和语义交错推进。

本讲后续的重点，就是如何把第二种做法系统化，包括自底向上的属性计算、翻译方案，以及可自动生成的表驱动实现。

## 2. S 属性文法的自底向上语义分析

最容易和自底向上分析结合的是 S 属性文法，因为它只含综合属性。信息从子结点流向父结点，这和归约方向天然一致。

对括号文法：

```text
[0] S' -> S EOF
[1] S  -> epsilon
[2] S  -> [ S1 ] S2
```

可以用下面的属性规则统计匹配括号对数：

$$
S'.cnt = S.cnt
$$

$$
S.cnt = 0
$$

$$
S.cnt = 1 + S_1.cnt + S_2.cnt
$$

在移进-归约分析过程中，每个非终结符在分析栈上都可以带着自己的综合属性。当执行归约 `A -> B1 ... Bk` 时，`B1 ... Bk` 的属性值已经在栈上，因此分析器可以立刻计算 `A` 的属性，再把结果写回归约后的栈槽位。

这就得到 S 属性文法的通用自底向上方案：

1. 正常移进语法符号；
2. 对已经完成的非终结符，在栈上附带综合属性；
3. 归约时，根据右部属性计算左部属性；
4. 将结果存入新的栈项。

这是最干净的一类情况，因为语法分析顺序和属性流向顺序几乎完全一致。

## 3. 继承属性如何放进自底向上分析

L 属性文法更难，因为它不仅有综合属性，还有继承属性，而继承属性往往沿着自顶向下或从左到右的方向流动，看起来和自底向上分析不匹配。

关键观察是：如果原始文法本身是 LL 的，那么当自底向上分析器从左到右移进 token 时，它仍然能够利用已经看到的上下文，模拟出 LL 分析本会做出的展开决策。这样一来，分析器就能在“未来归约会用到的位置”提前预留栈槽，把继承属性先放进去。

对于同一个括号语言，可以用继承属性和综合属性来统计嵌套深度：

$$
S.i = 0
$$

$$
S.s = S.i
$$

$$
S_1.i = S.i + 1
$$

$$
S_2.i = S_1.s
$$

$$
S.s = S_2.s
$$

![继承属性可以提前存放在将来参与归约的位置下方](lec10_materials/bottom_up_inherited_attributes.png)

这套技巧的要点是：把分析栈看成“未来继承属性的暂存区”。在某个非终结符稍后会被归约之前，就把它需要的继承属性保存在那个未来归约位置下方、静态可确定的偏移处。

:::remark 📝 问题：继承属性看起来是自顶向下流动的，为什么还能在自底向上分析里处理？
问题是：**如果继承属性依赖父结点或左兄弟的信息，移进-归约分析器怎么可能提前算出来？** 关键在于，左到右的上下文已经足够确定 LL 式的展开决策。一旦分析器知道接下来会展开哪个非终结符出现，它就能提前把所需的继承属性写进将来可以访问到的栈位置。
:::

为了把这些位置写清楚，讲义引入了 `L`、`M`、`N` 这类标记非终结符。它们并不是源程序语法的一部分，而是专门用来标记“继承属性应该在这里写入”的位置。

![标记非终结符让一个 L 属性的 LL 文法可以被移进-归约方式实现](lec10_materials/l_attributed_lr_implementation.png)

从纯属性文法的角度看，这种做法略带工程化味道；但从实现角度看，它非常实用，并且自然过渡到后面的语法制导翻译方案。

## 4. 从 SDD 到 SDT

属性文法，也就是 **Syntax-Directed Definition (SDD)**，描述的是“应该计算什么”。而 **SDT** 描述的是“应该在什么时候执行对应的语义代码”。

![SDT 把属性规则改写为可执行的语义动作片段，并插入到产生式右侧](lec10_materials/sdt_overview.png)

这里有一句非常值得保留：**Syntax-Directed Translation, SDT** 会把属性计算规则改写成用花括号 `{}` 包围的程序片段，并把它们插入到产生式右侧合适的位置。

和 SDD 相比，SDT 多做了两件事：

- 它把声明式的属性方程改写成可执行的语义动作；
- 它把执行顺序显式化了。

可以把 SDT 理解为：在深度优先遍历语法分析树时，语义动作究竟应该在哪个时刻触发。

对 S 属性文法来说，动作通常放在产生式最右端，因为只有等整个右部都分析完成，所需的子属性才全部可用。对 L 属性文法来说，动作也可以插在中间，紧挨着它要为之准备继承属性的那个符号之前。

:::tip 💡 问题：既然属性方程已经写清楚了，为什么还要从 SDD 改写成 SDT？
问题是：**如果属性方程已经说明要算什么，为什么还要再引入语义动作？** 因为实现需要一个明确的执行时序。SDD 只给出了依赖关系，而 SDT 把这些依赖关系落实成在语法分析或树遍历过程中哪个时刻执行哪段代码。
:::

讲义还强调，SDT 的动作不一定只用来计算属性，也可以直接产生副作用，比如打印翻译结果。

![SDT 的通用实现方式是把语义动作视为语法分析树中的特殊孩子结点](lec10_materials/sdt_embedded_action_tree.png)

这种通用树实现方式是：

1. 建树时先忽略动作结点；
2. 再把动作补回到对应产生式结点的孩子列表中；
3. 对整棵树做前序遍历；
4. 一旦访问到动作结点，就立刻执行动作。

即使后面我们转向表驱动分析，这个视角依然重要，因为它解释了动作顺序本来应该是什么样。

## 5. 基于 LL 文法的表驱动语义分析

普通的表驱动 LL 分析器，栈里只存终结符和非终结符。要支持语义动作，栈里就不能只放语法符号。

本讲为语法分析栈增加了两种记录：

- 综合记录：保存非终结符的综合属性；
- 动作记录：保存要执行的代码，以及动作执行时所需的属性上下文。

![LL 风格的表驱动语义分析需要同时维护综合记录和动作记录](lec10_materials/ll_semantic_stack_records.png)

对于 S 属性或 L 属性的 LL 文法，把 SDD 改写为 SDT 时，可以遵循一条很清晰的规则：

- 计算某个非终结符继承属性的动作，插在该非终结符出现之前；
- 计算产生式左侧综合属性的动作，插在产生式最右端。

当分析器展开产生式 `A -> B1 ... Bk` 时，仍然像 обычной LL 分析那样逆序压栈。但区别在于：

- 如果 `Bi` 是非终结符，就把 `Bi` 和它的综合记录一起压栈；
- 如果 `Bi` 是语义动作 `act`，就压入动作记录；
- 终结符仍然按普通方式压栈。

![LL(1) 语义分析驱动程序会在记录弹出到栈顶时立即执行它们](lec10_materials/ll_semantic_driver.png)

驱动循环于是变成：

1. 如果栈顶是动作记录或综合记录，就执行它并弹出；
2. 如果栈顶是非终结符，就查 LL 表并展开；
3. 如果栈顶是与当前 token 匹配的终结符，就同时消费它们；
4. 否则报错。

:::tip 💡 问题：没有递归调用栈之后，表驱动 LL 分析器怎样支持语义动作和继承属性？
问题是：**递归下降里靠函数调用自然传递属性，表驱动实现里这些信息放在哪？动作又什么时候执行？** 答案是把它们显式放进栈记录里。动作记录同时保存代码和它需要的属性副本；综合记录则把已经完成的属性值放在对应非终结符下方。
:::

这种方法的漂亮之处在于：递归控制结构消失了，但原本的语义执行顺序以一种机械而稳定的形式保留下来了。

## 6. 基于 LR 文法的表驱动语义分析

LR 情况又不一样。自底向上的分析器通常要等到归约发生时，才能确定当前到底匹配的是哪条产生式。这会立刻带来一个翻译方案设计约束。

讲义给出的解决办法非常明确：对基于 LR 的表驱动语义分析，语义动作应当放在产生式最右侧。这样语义动作就只会在归约时执行，而归约时产生式已经确定。

![LR 风格的语义分析栈同时保存自动机状态、符号和已计算属性](lec10_materials/lr_semantic_shift_reduce.png)

这和 S 属性文法天然契合：

- 正常移进 token 和状态；
- 对已完成的非终结符，在栈项里保存属性；
- 当归约 `A -> alpha` 时，用 `alpha` 的属性执行语义动作；
- 弹出右部、压入 `GOTO` 后的状态，并把 `A` 的综合属性写入新栈项。

讲义还展示了一个更显式的 SDT 版本：语义动作可以直接操作 `stack` 和 `top`，从而把运行机制写得非常直白。

而对于那些以 LL 文法为基础的 L 属性文法，则可以继续利用前面介绍的标记非终结符技巧，把它们编译成显式操作分析栈的 LR 风格 SDT。

:::warn ⚠️ 问题：为什么基于 LR 的 SDT 一般要求把语义动作放在最右边？
问题是：**为什么移进-归约分析器不能像 LL 那样，随便在产生式中间执行动作？** 因为在归约之前，分析器往往还不能确定当前栈片段一定属于哪一个具体产生式出现。把动作放到最右端可以避开这个不确定性：等到归约发生时，产生式已知，右部信息齐全，动作才可以安全执行。
:::

这正是一个很典型的例子：实现机制会反过来约束我们接受哪种翻译方案写法。

## 7. 类型表达式与类型等价

本讲后半段从属性计算机制转向类型系统。**type expression** 指的是用结构化方式表示变量、常量、表达式、语句等程序成分类型的写法。

讲义给出的基本构造包括：

- `bool`、`char`、`int`、`float`、`void` 等基本类型；
- 面向对象语言中的类名；
- 数组类型；
- 记录类型；
- 函数类型；
- 笛卡尔积，可用于元组和参数列表；
- 取值为类型表达式的变量。

函数类型和元组式类型分别写成：

$$
s \to t
$$

以及：

$$
s \times t
$$

![一个 C 结构体既可以写成带字段名的类型表达式，也可以写成忽略字段名的结构表示](lec10_materials/struct_type_expression.png)

例如，下面这个 C 结构体：

```c
struct {
  int no;
  char name[20];
}
```

可以表示为：

$$
\operatorname{record}(\{no:int, name:\operatorname{array}(20, char)\})
$$

或者在忽略字段名时写成：

$$
int \times \operatorname{array}(20, char)
$$

类型等价讨论的是：两个类型表达式什么时候算“同一个类型”。讲义区分了两种主要思想：

- **按名字等价**：因为名字相同，或被定义为等价名字，所以类型相同；
- **按结构等价**：即使名字不同，只要内部结构完全相同，也算同一类型。

![类型等价主要分为按名字等价和按结构等价](lec10_materials/type_equivalence_modes.png)

结构等价的典型例子是：

$$
\{x:int, y:int\} \equiv \{y:int, x:int\}
$$

讲义还特别强调：按名字等价一定意味着按结构等价，但反过来不一定成立。

## 8. 类型检查、类型转换、重载、推导与多态

讲义把 **type system** 定义为：给程序每个组成部分赋予类型表达式，并通过一组逻辑规则约束这些类型。这里有个关键词非常重要：**soundness**。它表示类型系统可以保证某一类运行时类型错误不会发生。

类型检查规则被分成两种互补风格：

- **type synthesis**：根据子表达式的类型构造出当前表达式的类型；
- **type inference**：根据表达式的使用方式反推出缺失的类型。

![讲义把类型综合、类型推导和双向类型系统并列讨论](lec10_materials/type_rule_categories.png)

最基本的类型综合规则是：

如果 \(f : s \to t\)，且 \(x : s\)，那么：

$$
f(x) : t
$$

对应的推导视角则是：如果 `f(x)` 是一个表达式，那么对某些类型 `alpha` 和 `beta`，有：

$$
f : \alpha \to \beta,\quad x : \alpha
$$

Bidirectional Typing 就是把这两者结合起来：能综合时就综合，缺少信息时就推导。

讲义还回顾了数值类型的隐式与显式转换。

![拓宽保留数值范围信息，而窄化可能丢失范围信息](lec10_materials/type_widening_narrowing.png)

用 Java 风格术语来说：

- **widening**：保留原有数值范围信息，但可能损失精度；
- **narrowing**：可能丢失数值范围信息，因此通常要求显式转换。

重载又带来了新的问题：同一个符号的含义会依赖上下文。

![重载函数通过实参类型来确定采用哪一个函数类型](lec10_materials/overloaded_function_typing.png)

如果 \(f\) 可能有类型 \(s_i \to t_i\)（其中 \(1 \le i \le n\)，且 \(s_i \ne s_j\) 当 \(i \ne j\)），并且 \(x : s_k\)，那么：

$$
f(x) : t_k
$$

类型推导则直接回应了“类型标注太多”这个现实问题。

![类型推导可以被形式化为类型方程组，并通过求解得到结果](lec10_materials/type_inference_equations.png)

对例子：

```text
val null : list(int) -> bool
val tl   : list(int) -> list(int)
fun length(x) =
  if null(x)
  then 0
  else length(tl(x)) + 1
```

讲义为各个子表达式引入 `alpha_1` 到 `alpha_8` 这样的类型变量，并推导出方程，例如：

$$
\alpha_2 \to \alpha_3 = list(int) \to bool
$$

$$
\alpha_2 \to \alpha_5 = list(int) \to list(int)
$$

$$
\alpha_1 = \alpha_2 \to \alpha_8
$$

解完整个方程组之后得到：

$$
\alpha_1 = list(int) \to int
$$

:::remark 📝 问题：类型化语言往往要求很多类型标注，类型推导究竟帮了什么？
问题是：**既然带类型语言更安全，为什么程序员仍然常常抱怨它们啰嗦？** 因为显式类型标注可能很多。类型推导把这些缺失类型先当作未知量，再根据程序结构生成方程，最后通过求解恢复出与程序一致的类型。
:::

最后一步是参数多态。

![泛化把一个具体类型实例提升成可在多个类型上复用的多态函数模式](lec10_materials/parametric_polymorphism_generalization.png)

我们不再把：

$$
list(int) \to bool
$$

和：

$$
list(int) \to list(int)
$$

固定死，而是把它们泛化为：

$$
\forall \beta.\ list(\beta) \to bool
$$

以及：

$$
\forall \beta.\ list(\beta) \to list(\beta)
$$

于是最终得到：

$$
length : \forall \beta_1.\ list(\beta_1) \to int
$$

:::tip 💡 问题：既然方程已经解出来了，为什么多态还需要“泛化”这一步？
问题是：**局部约束求解完之后，结果不是已经够了吗？** 还不够。多态函数必须能在很多具体类型上重复使用。局部求解得到的只是一次具体实例，而泛化会把这个结果提升成一个带全称量化的类型模式，让后续不同调用点都能再次实例化它。
:::

## 9. 类型系统为什么重要

讲义最后用一段很紧凑的历史和设计讨论作结。

早期的 B 语言本质上是无类型的：参数没有显式类型，函数没有返回类型声明，外部函数没有签名，`auto` 也只是“预留存储空间”的意思。这意味着编译期检查能力很弱，很多责任被推到了运行时。

![从 B 到 C 的演化很好地说明了显式类型如何改进检查能力与性能](lec10_materials/b_to_c_typed_evolution.png)

到了 C 语言，参数类型、返回类型、局部变量类型，以及像下面这样的函数签名都变得显式：

```c
extern int putchar(int);
void printn(int n, int b) { ... }
```

这不仅提高了安全性，也提高了性能。讲义最后总结了类型系统的重要价值：

- 它能检查错误，比如函数调用不满足签名、对不合适的值执行不合法操作；
- 它能提供文档；
- 它能帮助编译优化，例如去掉不必要的运行时检查、优化内存布局；
- 它能提升语言安全性；
- 它能支撑抽象机制，如带类型的泛型、面向对象和函数式编程。

最后一个设计层面的结论尤其重要：语言设计和类型系统设计应当并行推进。无类型语言在设计过程中很容易引入日后难以做健全类型检查的特性；而带类型语言则需要在语法层面就考虑类型标注和类型驱动结构是否足够清晰。

## 10. Exam Review

### 关键定义

- **semantic elaboration**：把已解析的语法翻译成更显式的语义或中间表示。
- **S 属性文法**：只使用综合属性的属性文法。
- **L 属性文法**：每个继承属性只依赖父结点和左侧符号信息的属性文法。
- **Syntax-Directed Definition (SDD)**：声明式的属性文法规约。
- **Syntax-Directed Translation (SDT)**：**把属性计算改写成插入产生式中的可执行语义动作**。
- **type expression**：程序构造类型的结构化表示。
- **type equivalence**：类型按名字或按结构意义上的相等关系。
- **type synthesis**：**根据子表达式类型构造表达式类型**。
- **type inference**：根据使用方式恢复缺失类型。
- **soundness**：保证某一类运行时类型错误不会发生。
- **parametric polymorphism**：同一个函数可以在多个由类型变量刻画的类型上运行。

### 必须会解释的机制

1. 为什么综合属性天然适合自底向上分析。
2. 继承属性如何通过“未来归约位置下方的栈槽位”来保存。
3. 标记非终结符如何把一个 L 属性的 LL 文法变成可操作的 LR 式实现。
4. 为什么说 SDT 是对 SDD 的执行顺序细化。
5. LL 表驱动分析如何同时保存综合记录和动作记录。
6. 为什么 LR 表驱动分析更偏好最右侧语义动作。
7. 类型表达式如何表示数组、记录、函数类型和元组。
8. 按名字等价和按结构等价有什么区别。
9. 类型综合和类型推导有什么不同。
10. 类型方程如何被求解，从而得到推导出的函数类型。
11. 为什么多态推导在求解之后还需要泛化。
12. 为什么从 B 到 C 的历史演进能说明静态类型的价值。

### 简答模板

- **继承属性如何放进自底向上分析**：说明分析器利用左到右上下文预留栈槽位，并常借助标记非终结符写入未来会用到的继承值。
- **为什么需要 SDT**：说明 SDT 把属性依赖落实成可执行的动作位置，从而明确了计算顺序。
- **LL 语义分析驱动程序**：提到综合记录、动作记录，以及它们被弹到栈顶时立即执行。
- **LR 语义分析驱动程序**：提到只有归约时才能确定产生式，因此把动作放在最右侧最安全。
- **类型综合规则**：如果 `f : s -> t` 且 `x : s`，那么 `f(x) : t`。
- **类型推导**：把缺失类型替换成变量，根据语法结构生成方程，求解后若需要多态再做泛化。

### 常见误区

- 认为继承属性完全不能和自底向上分析结合。实际上可以，只是栈布局必须提前安排好。
- 把 SDT 简单理解成“带代码的 SDD”。真正关键的差异在于它还固定了执行顺序。
- 忘记 LL 表驱动语义分析需要显式动作记录，而不是只保留非终结符。
- 在 LR 翻译方案里随意把动作塞到产生式中间，却没有保证分析器当时已经知道具体产生式。
- 混淆按结构等价与按名字等价。
- 把所有自动得到类型的情况都叫“类型综合”，而忽略其中有些其实是从未知量出发的真正类型推导。
- 只解完类型方程，却忘了为了多态复用还要做泛化。
