# Lec1：编译器导论与语言处理流程

编译器把源程序翻译成目标程序，同时尽量保留程序的语义。一个典型编译器会经历词法分析、语法分析、语义分析、中间表示生成、优化和目标代码生成等阶段。

## 从字符到结构

源代码最初只是字符序列。词法分析器将字符聚合为 token，语法分析器再依据文法构造抽象语法树。对于表达式 `a + b * c`，乘法节点通常会位于加法节点的子树中，以表达优先级。

行内公式示例：若 `G` 是一个上下文无关文法，推导关系可写作 $\alpha A \beta \Rightarrow \alpha \gamma \beta$。

$$
E \rightarrow E + T \mid T,\qquad
T \rightarrow T * F \mid F,\qquad
F \rightarrow (E) \mid id
$$

:::remark 📝 抽象语法树为什么重要
抽象语法树删去了括号、分号等只服务语法的细节，保留程序结构。后续语义检查、类型推导和 IR 生成通常都围绕 AST 展开。
:::

## 编译器流水线

- 词法分析：字符流变为 token 流。
- 语法分析：token 流变为语法树或抽象语法树。
- 语义分析：检查名字、作用域、类型与控制流约束。
- 中间表示：把语言相关结构转为更利于优化的 IR。
- 代码生成：选择指令、分配寄存器并输出目标代码。

:::tip 💡 阅读建议
学习每个阶段时，优先问两个问题：输入是什么，输出是什么。这样可以更容易把局部算法放回完整编译流程。
:::

## 一个极小的 IR 片段

```text
t1 = b * c
t2 = a + t1
return t2
```

> IR 的目标不是“更像机器码”，而是让分析和变换更稳定、更可控。

:::warn ⚠️ 常见混淆
语法正确不代表语义正确。例如 `x + true` 可能通过语法分析，但会在类型检查阶段报错。
:::

:::error ⛔ 易错点
不要把具体语法树和抽象语法树混为一谈。前者通常保留完整推导过程，后者更关注程序的语义结构。
:::
