# Lec12: Advanced Topics in Intermediate Representations

## 1. What This Lecture Adds Beyond Basic IR

Lec5 introduced the main IR families: AST, DAG, CFG, three-address code, and SSA. This lecture moves from “what these IRs are” to “how to generate them systematically” during semantic analysis and parsing. The three main threads are:

- turning syntax-directed translation into executable IR-generation rules;
- handling arrays, control flow, and short-circuit boolean expressions without losing correctness;
- constructing SSA-form IR, first in a simple way and then in progressively better ways.

The central mindset is that IR generation is not a separate afterthought. Once we know what information each construct must contribute, we can attach the corresponding actions directly to grammar productions.

## 2. SDT Revisited: Batch Construction vs On-the-Fly Emission

**Syntax-Directed Translation (SDT)** rewrites attribute computations as code fragments `{ ... }` inserted into productions. Operationally, those fragments are executed at specific moments during a depth-first traversal of the parse tree.

For expressions and assignments, a classic batched scheme keeps two synthesized attributes:

- `E.addr`: the address holding the value of `E`;
- `E.code`: the three-address code generated for `E`.

The standard helpers are:

- `genvar(x)`: generate the address of a program variable;
- `gencst(c)`: generate the address of a constant;
- `gentmp()`: generate the address of a fresh temporary.

For example, the translation of `E1 + E2` conceptually does three things:

1. generate code for `E1`;
2. generate code for `E2`;
3. append one more instruction that stores the sum into a fresh temporary.

That is why the lecture uses code concatenation with `||`. But concatenating strings of code is mechanically redundant, so the lecture immediately moves to a better idea: emit instructions as side effects.

In the on-the-fly variant, we still compute `E.addr`, but instead of building `E.code` as a string, we directly call:

```text
emit("{t} = {a1} + {a2}")
```

The parse still determines the order in which instructions appear, but the generated IR is now incremental rather than assembled at the end.

:::remark 📝 Question: why is on-the-fly translation better than building `E.code` strings first?
The real issue is not expressiveness but redundancy. Concatenating `E1.code || E2.code || ...` repeatedly rebuilds intermediate strings that we will immediately flatten again. With `emit(...)`, the SDT still follows the parse-tree order, but instruction generation becomes direct, cheaper, and closer to how a real frontend streams IR into a builder.
:::

## 3. Translating Declarations, Blocks, and Structured Types

Once translation moves beyond arithmetic expressions, the frontend must track scope and type information as carefully as it tracks temporary values.

For declarations such as:

```text
S -> T ID ;
```

the key semantic action is:

```text
insert(ID.lexeme, T.type)
```

This inserts the declared name and its type into the current symbol table scope.

Blocks require explicit scope management:

```text
S -> { L }
```

with actions equivalent to:

```text
push_scope()
...
pop_scope()
```

and statement lists concatenate the code generated by their components. The same scope-stack idea extends naturally to `struct` declarations: parse the field declarations inside a fresh scope, then package the popped scope into a struct type.

Arrays are more interesting because the frontend must compute both the type expression and the total width. The lecture uses inherited attributes `C.t` and `C.w` to thread the current element type and element width through the array suffixes.

![Array type construction records both nested element type and total width](lec12_materials/array_type_width_rules.png)

The essential rules are:

$$
TB \to int:\quad TB.type = int,\quad TB.width = 4
$$

$$
TB \to double:\quad TB.type = double,\quad TB.width = 8
$$

$$
C \to \epsilon:\quad C.type = C.t,\quad C.width = C.w
$$

$$
C \to [INT]\ C_1:\quad C.type = array(INT.intval,\ C_1.type),\quad C.width = INT.intval \times C_1.width
$$

So `int[2][3]` is processed inside out:

- base type `int`, width `4`;
- `[3]` gives `array(3, int)`, width `12`;
- `[2]` gives `array(2, array(3, int))`, width `24`.

This width information is not decoration. Later code generation needs it to allocate storage and to compute offsets for array accesses.

## 4. Computing Array Addresses Correctly

The array-reference grammar is:

```text
R ::= R[E] | ID[E]
E ::= ... | R
S ::= ... | R = E ;
```

The key question is: how do we turn a nested reference into an offset relative to the array base address?

For one-dimensional arrays stored contiguously:

$$
\mathrm{addr}(A[i]) = base + i \times w
$$

where `w` is the width of one element.

For a two-dimensional row-major array:

$$
\mathrm{addr}(A[i_1][i_2]) = base + i_1 \times w_1 + i_2 \times w_2
$$

For a `k`-dimensional row-major array, let `n_j` be the number of elements along dimension `j`, and let `w_j` be the width of one element of the `j`-th subarray level. Then:

$$
w_k = w
$$

$$
w_{j-1} = n_j \times w_j
$$

$$
\mathrm{addr}(A[i_1][i_2]\cdots[i_k]) = base + i_1 w_1 + i_2 w_2 + \cdots + i_k w_k
$$

An equivalent nested computation is:

$$
\mathrm{addr}(A[i_1][i_2]\cdots[i_k]) = base + ((\cdots((i_1 \times n_2 + i_2)\times n_3 + i_3)\cdots)\times n_k + i_k)\times w
$$

![Row-major addressing converts a multidimensional reference into a single linearized offset](lec12_materials/row_major_array_addressing.png)

The lecture also reminds us that indexing does not always start at `0`. In a Pascal-style array declared over `[low..high]`, the address formula becomes:

$$
\mathrm{addr}(a[i]) = base + (i - low)\times w
$$

Since `low` is known from the type, `base - low \times w` can often be precomputed.

:::tip 💡 Question: why do compilers care so much about array width and storage order?
Because a multidimensional array reference is not primitive at the machine level. The backend only knows addresses and offsets. Width tells us how many bytes one element or subarray occupies, and storage order tells us which dimension contributes the larger stride. Without those two facts, `a[i][j]` is not compilable.
:::

## 5. Translating Array Reads and Writes

To translate array references incrementally, the lecture gives each array-reference node `R` three synthesized attributes:

- `R.addr`: an address that stores the current offset index;
- `R.base`: the base address of the whole array;
- `R.type`: the type of the current subarray or element.

![The array-reference SDT separates base address, offset, and current subarray type](lec12_materials/array_reference_translation.png)

The base case:

```text
R -> ID[E]
```

means:

- the first index expression already determines the initial offset;
- `R.base` is the address of the array variable itself;
- `R.type` becomes the element type of the declared array type.

The recursive case:

```text
R -> R1[E]
```

extends the previous offset by multiplying the earlier offset with the length of the next dimension, then adding the new index.

When an array reference is used as an expression:

```text
E -> R
```

we still need one more step: multiply the logical offset by the width of the final element type, then load from `R.base[...]`.

That is why reading `a[i][j]` for `a : int[2][3]` becomes:

```text
t0 = i * 3
t1 = t0 + j
t2 = t1 * 4
t3 = a[t2]
```

Writing is analogous. For:

```text
a[i * j + k][l] = b[m - n];
```

the generated three-address code is:

```text
t0 = i * j
t1 = t0 + k
t2 = t1 * 3
t3 = t2 + l
t4 = t3 * 4
t5 = m - n
t6 = t5 * 4
t7 = b[t6]
a[t4] = t7
```

The deeper lesson is that nested array syntax is gradually lowered into a single linear offset plus one final memory operation.

## 6. Backpatching Boolean Expressions

Short-circuit boolean translation creates jumps before all their targets are known. That is exactly why **backpatching** is needed.

Instead of inherited target labels such as `B.true` and `B.false`, the lecture uses synthesized lists:

- `B.truelist`: jumps whose destination should be the code reached when `B` is true;
- `B.falselist`: jumps whose destination should be the code reached when `B` is false.

The helper operations are:

- `makelist(i)`: create a singleton list containing instruction `i`;
- `merge(p1, p2)`: concatenate two pending-jump lists;
- `backpatch(p, i)`: fill every placeholder in list `p` with target instruction `i`.

The translation is driven by a global `nextinstr` counter in an incremental code generator.

![Backpatching for && and || preserves short-circuit evaluation without a second symbolic-label pass](lec12_materials/backpatch_boolean_short_circuit.png)

The important cases are:

- `true`: emit `goto _`, and put that instruction into `truelist`;
- `false`: emit `goto _`, and put that instruction into `falselist`;
- comparison such as `E1 == E2`: emit one conditional jump for the true case and one unconditional jump for the false case;
- `B1 && B2`: backpatch `B1.truelist` to the beginning of `B2`;
- `B1 || B2`: backpatch `B1.falselist` to the beginning of `B2`;
- `!B1`: swap `truelist` and `falselist`.

For:

```text
x < 100 || (x > 200 && x != y)
```

the lecture shows the generated skeleton:

```text
100: if x < 100 goto _
101: goto _
102: if x > 200 goto _
103: goto _
104: if x != y goto _
105: goto _
```

and the final pending lists become:

- overall `truelist = {100, 104}`;
- overall `falselist = {103, 105}`.

Those lists are exactly what later statement translation will resolve.

:::remark 📝 Question: why not decide every jump target immediately when translating a boolean expression?
Because short-circuit translation discovers control flow before it has finished generating all successor code. When we emit the jump caused by `B`, the code for “what comes next if `B` is false” may not exist yet. Backpatching lets us generate jumps in one pass, keep their destinations open temporarily, and fill them in as soon as the destination becomes known.
:::

## 7. Backpatching Control-Flow Statements

Once booleans use `truelist` and `falselist`, statements need one more synthesized attribute:

- `S.nextlist`: jumps that should go to the instruction immediately following statement `S`.

Statement lists similarly use `L.nextlist`.

This yields a clean one-pass translation strategy:

- simple assignments and declarations contribute no pending exits;
- a block inherits the `nextlist` of its internal statement list;
- in a statement sequence, the earlier statement's `nextlist` is backpatched to the start of the later statement;
- `if(B) S1` backpatches `B.truelist` into `S1`, and merges `B.falselist` with `S1.nextlist`;
- `if(B) S1 else S2` needs one extra jump to skip over the `else` arm after the `then` arm finishes;
- `while(B) S1` backpatches `B.truelist` into the body, backpatches the body's `nextlist` to the loop test, emits a jump back to the loop head, and uses `B.falselist` as the loop exit.

For:

```text
if (x < 100 || (x > 200 && x != y))
  x = 0;
```

the condition's `truelist` is backpatched to the assignment, while the remaining false exits become the statement's `nextlist`.

The larger nested example in the lecture also shows a practical point: one-pass translation often produces redundant `goto`s. That is acceptable. Correctness comes first; later optimization passes can simplify control flow.

:::tip 💡 Question: what does “one-pass” really mean here?
It does not mean we magically know every target in advance. It means we can generate all instructions in a single forward traversal, while representing unknown targets as pending lists. Backpatching is the bookkeeping trick that makes one-pass control-flow generation possible.
:::

## 8. Generating CFG-Form IR Directly

Instead of producing a flat instruction list and recovering basic blocks later, we can build a control-flow graph directly.

**A basic block is a linear sequence of instructions, terminated by a jump or a return, in which control enters only at the first instruction.**

The lecture assumes a small CFG-building interface:

- `emit(instr)`: append an instruction to the current basic block;
- `new_bb()`: create a fresh basic block;
- `set_bb(bb)`: switch the current insertion point to block `bb`.

Two structural constraints matter:

- jumps and returns must appear only at the end of a basic block;
- each function should have a unique entry block.

![Short-circuit boolean translation can target whole basic blocks directly instead of pending flat instruction numbers](lec12_materials/cfg_short_circuit_translation.png)

In this representation, boolean translation uses inherited block targets:

- `B.true`: the block entered when the condition succeeds;
- `B.false`: the block entered when the condition fails.

Then:

- `true` emits `goto B.true`;
- `false` emits `goto B.false`;
- comparisons emit full conditional branches;
- `B1 && B2` creates a new block for the point where `B2` is evaluated if `B1` succeeds;
- `B1 || B2` creates a new block for the point where `B2` is evaluated if `B1` fails.

Similarly, `if` and `while` allocate explicit blocks for:

- the true branch;
- the false branch;
- the join block after the statement;
- the loop body and loop exit.

This form is often easier to connect to later SSA construction, because branch structure is explicit from the beginning rather than reconstructed after the fact.

:::warn ⚠️ Question: how do we support `break` and `continue` in direct CFG generation?
Treat them as statements that depend on loop context. Each loop should thread two inherited targets downward: the loop-exit block and the loop-continue target (usually the condition block or loop head). Then `break` simply emits `goto exit`, and `continue` emits `goto continue_target`. The point is that they are not special expressions; they are context-sensitive jumps.
:::

## 9. SSA Refresher: Phi Functions and Block Parameters

**Static Single Assignment (SSA) form means every assignment uses a distinct variable name, and every use of a name has a unique definition.**

In straight-line code, SSA is just systematic renaming. For example:

```text
p = a + b
q = p - c
p = q * d
p = e - p
q = p + q
```

becomes:

```text
p1 = a + b
q1 = p1 - c
p2 = q1 * d
p3 = e - p2
q2 = p3 + q1
```

The interesting part appears at control-flow merges. In traditional SSA, a merge is represented by a phi-function:

$$
x_3 = \phi(x_1,\ x_2)
$$

The lecture also shows an equivalent block-parameter style, where the join block explicitly takes incoming values as parameters.

![Phi-functions and block parameters express the same merge information in two different surface forms](lec12_materials/ssa_phi_and_block_parameters.png)

The same idea extends to loops. A loop header may receive the “initial” values from the preheader and the “updated” values from the back edge. In phi-form, those are `phi` nodes; in block-parameter form, those are loop-header parameters.

:::remark 📝 Question: why are block parameters a valid replacement for phi-functions?
Because both mechanisms express the same merge semantics: a use at the join point should see the value coming from the predecessor that actually transferred control there. A phi-function says this inside the block; a block parameter says the same thing at the control-flow edge. The information is equivalent, only the notation changes.
:::

## 10. From Ordinary Variables to Naive SSA via Memory Cells

A direct obstacle to SSA generation is that source-level variables are assigned many times. One simple workaround is to stop treating them as SSA values at first and instead give each program variable a memory cell.

Then variable updates become explicit `store`s, and variable uses become explicit `load`s. The lecture uses operations such as:

- `alloc`
- `load`
- `store`

For example:

```text
a = alloc
store 1, a
b = alloc
store 1, b
c = alloc
t0 = load a
t1 = load b
t2 = t0 + t1
store t2, c
t3 = load b
store t3, a
t4 = load c
store t4, b
```

This representation is already close to SSA, because the temporaries `t0`, `t1`, `t2`, ... are single-assignment values. The remaining mutable state has been pushed into explicit memory operations.

The downside is obvious: many loads and stores are artifacts of the translation rather than real data dependencies. A smarter compiler may track the current address or current value associated with a source variable in the symbol table and avoid unnecessary stores until they are actually needed.

:::remark 📝 Question: why is the memory-cell method considered naive even though it is correct?
Because correctness comes at the cost of precision. Every reassignment is lowered to memory traffic, so the IR forgets many value-flow facts that were still obvious in the source program. Later passes then have to recover them with mem2reg-style promotion or other optimization. It works, but it starts from a noisier IR than necessary.
:::

## 11. Two-Pass SSA Construction with Block Parameters

The lecture then develops a cleaner SSA-generation strategy in block-parameter form.

The idea is:

1. perform a coarse pass that gives each basic block all variables that may be needed at block entry;
2. within each block, use local value numbering to rename computations into SSA style;
3. perform a fine-grained second pass that removes redundant block parameters.

The running example is the `pow(b, e)` loop:

```text
pow(b, e):
  r = 1
  goto loop
loop:
  if e > 0 then goto body else goto done
body:
  r = r * b
  e = e - 1
  goto loop
done:
  return r
```

After the coarse pass and local renaming, we may get:

```text
pow(b0, e0):
  r0 = 1
  goto loop(b0, e0, r0)
loop(b1, e1, r1):
  if e1 > 0
    then goto body(b1, e1, r1)
    else goto done(b1, e1, r1)
body(b2, e2, r2):
  r3 = r2 * b2
  e3 = e2 - 1
  goto loop(b2, e3, r3)
done(b3, e4, r4):
  return r4
```

Now comes the refinement rule. If a block parameter always receives either the same value or a value that can be identified with another parameter on every incoming edge, then that parameter is redundant and can be removed.

![The second pass removes redundant block parameters once all incoming arguments are known](lec12_materials/two_pass_ssa_parameter_elimination.png)

In the running example, `b` is loop-invariant and does not need to stay as a loop parameter forever, so the refined result becomes:

```text
pow(b0, e0):
  r0 = 1
  goto loop(e0, r0)
loop(e1, r1):
  if e1 > 0
    then goto body()
    else goto done()
body():
  r3 = r1 * b0
  e3 = e1 - 1
  goto loop(e3, r3)
done():
  return r1
```

This method is systematic and easy to reason about: over-approximate first, then prune.

:::tip 💡 Question: why is it safe to remove a block parameter in the second pass?
Because the second pass only removes a parameter after checking all incoming edges. If every jump to block `l` passes either the same argument or an argument equivalent to some other surviving parameter, then that removed parameter carries no independent information. Replacing it is a semantics-preserving simplification, not a guess.
:::

## 12. One-Pass SSA Construction

The lecture ends with a more direct algorithm that avoids the “add many parameters first, then prune” workflow.

The rule is:

- inside each basic block, use local value numbering to create SSA names;
- when a use of variable `x` is not defined in the current block:
  - if the block has one predecessor, recursively resolve `x` in that predecessor;
  - if the block has multiple predecessors, add `x` as a block parameter and recursively arrange actual arguments on incoming edges.

![One-pass SSA grows block parameters only when unresolved uses force them to appear](lec12_materials/one_pass_ssa_final_result.png)

Applied to `pow`, the final result is:

```text
pow(b0, e0):
  r0 = 1
  goto loop(e0, r0, b0)
loop(e2, r2, b2):
  if e2 > 0
    then goto body
    else goto done
body:
  r1 = r2 * b2
  e1 = e2 - 1
  goto loop(e1, r1, b2)
done:
  return r2
```

Compared with the two-pass method:

- the one-pass method introduces parameters only when demanded by unresolved uses;
- it often produces a tighter result immediately;
- but it is algorithmically trickier, because predecessor recursion and mutually dependent blocks must be handled carefully.

So the trade-off is classic compiler engineering:

- two-pass SSA is simpler and more obviously correct by construction;
- one-pass SSA is more direct and potentially cleaner, but requires more careful dependency handling.

:::remark 📝 Question: when should we prefer the one-pass method over the two-pass method?
Prefer it when your IR builder and CFG infrastructure can already support recursive predecessor queries and edge-argument updates cleanly. If you want a method that is easier to teach, debug, and prove correct, the two-pass approach is usually the calmer choice. If you want to avoid generating obviously redundant parameters in the first place, the one-pass approach is more elegant.
:::

## 13. Exam Review

### Key definitions

- **Syntax-Directed Translation (SDT)**: attach executable semantic actions to grammar productions and run them at specific traversal points.
- **Backpatching**: emit jumps with unknown destinations first, then fill their targets later once the destination instruction or block is known.
- **Basic block**: a linear instruction sequence with one entry point and a terminating jump or return.
- **SSA form**: each assignment defines a fresh name, and each use has a unique reaching definition.
- **Phi-function / block parameter**: two equivalent ways to describe value merging at CFG join points.

### Core mechanisms

- Expression SDT needs at least an address for the computed value and a strategy for emitting code.
- Array translation needs type width, dimension length, and storage order to compute linearized offsets.
- Boolean short-circuit translation is easiest to implement with `truelist`, `falselist`, `makelist`, `merge`, and `backpatch`.
- Statement translation extends the same idea with `nextlist`.
- CFG-form IR generation replaces pending instruction numbers with explicit basic blocks.
- SSA generation can be staged through memory cells, through a two-pass block-parameter construction, or through a one-pass predecessor-driven method.

### Short-answer templates

- Why do we need backpatching?
  Because short-circuit control flow emits jumps before all successor code is available, so we must record unresolved exits and fill them once the destination is generated.

- How do we translate multidimensional array accesses?
  Compute a row-major or column-major linearized offset from the indices, multiply the final logical offset by element width if needed, then add the base address and emit the load/store.

- What is the relationship between phi-functions and block parameters?
  They encode the same merge semantics: the value seen at a join point depends on which predecessor transferred control to that join.

- Why is alloc/load/store-based SSA generation only a baseline?
  Because it is correct but introduces unnecessary memory traffic, forcing later passes to recover register-like SSA values again.

- What is the main difference between two-pass and one-pass SSA construction?
  Two-pass over-approximates block parameters and then removes redundant ones; one-pass creates parameters only when unresolved uses force them.

### Common mistakes

- Confusing an array's logical index with its byte offset.
- Forgetting that `R.type` changes as nested array references peel one dimension at a time.
- Treating `&&` and `||` as if both operands were always evaluated.
- Thinking that backpatching is only for `if`; it is equally central for loops and statement sequences.
- Believing that SSA is just renaming; joins and loops are the real difficulty.
